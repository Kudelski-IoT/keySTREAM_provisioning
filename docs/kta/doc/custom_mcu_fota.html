FOTA Integration Guide

Note: FOTA integration and usage can be performed and evaluated only on the Trust Platform Design Suite (TPDS) application. Ensure you have the latest TPDS application installed, along with the keySTREAM Connect extension, to enable full FOTA functionality and testing.

Note: The FOTA Agent and FOTA Process modules are provided as part of the KTA package and do not require modification. Only the APIs listed below need to be implemented by you.

</blockquote>

Overview

FOTA integration phases

1. How FOTA Works — The Big Picture

The KTA library handles all communication with keySTREAM. You only implement the platform-specific parts — how to read your firmware version, how to download, and how to store state.

keySTREAM Cloud

│ Campaign available

│ (provided, │ │

That's it. Three touchpoints in fotaplatform.c, plus two storage functions in k_sal_fotastorage.c.

2. What You Must Implement

# Function File What It Does
1 fotaPlatformGetComponents() fotaplatform.c Return your firmware's current name and version
2 fotaStartInstalltation() fotaplatform.c Receive a download URL → download the binary → write to flash
3 fotaUpdateComponent() — (you call it) After download, tell KTA if it succeeded or failed
4 salFotaStorageWrite() k_sal_fotastorage.c Save FOTA campaign state to your NVM
5 salFotaStorageRead() k_sal_fotastorage.c Load FOTA campaign state from your NVM

fotaUpdateComponent() is already implemented by KTA. You only call it — you don't write it.

3. API Reference

3.1 <tt>fotaPlatformGetComponents()</tt>

KTA calls this at startup and when keySTREAM queries the device. Return the currently running firmware name and version.

TKFotaStatus fotaPlatformGetComponents(TComponent *xpComponents);

Fill these fields in xpComponents[0]:

Field Max Size Example
componentName 16 bytes "MY-APP"
componentNameLen — 6
componentVersion 16 bytes "2.1.0" (free-form string)
componentVersionLen — 5

Return: E_K_FOTA_SUCCESS or E_K_FOTA_ERROR

How you get the version is entirely up to you — read it from a struct in flash, a header constant, a file, a register, whatever works on your platform.

Minimal example:

TKFotaStatus fotaPlatformGetComponents(TComponent *xpComponents)
{
if (xpComponents == NULL)
return E_K_FOTA_ERROR;
// Read from wherever your firmware version lives
const char *name = "MY-APP";
const char *version = "2.1.0";
memcpy(xpComponents[0].componentName, name, strlen(name));
xpComponents[0].componentNameLen = strlen(name);
memcpy(xpComponents[0].componentVersion, version, strlen(version));
xpComponents[0].componentVersionLen = strlen(version);
return E_K_FOTA_SUCCESS;
}

3.2 <tt>fotaStartInstalltation()</tt>

KTA calls this when a FOTA campaign is ready. You receive the download URL and must handle the rest.

Note: The function name has a double-t (fotaStartInstalltation) — this is the actual symbol name in KTA.

void fotaStartInstalltation(const void *xpFotaMetadata,
size_t xFotaMetadataLen,
const TTargetComponent *xpComponents);

What you get from xpComponents:

Field Max Size Description
componentTargetName / componentTargetNameLen 16 bytes Name of component to update
componentTargetVersion / componentTargetVersionLen 16 bytes Target version
componentUrl / componentUrlLen 512 bytes HTTPS URL to download the firmware binary

What you must do:

  1. Extract the URL from xpComponents->componentUrl
  2. Download the binary using your network stack (WiFi, Ethernet, LTE — your choice)
  3. Write it to flash using your flash driver
  4. Call fotaUpdateComponent() when done (success or failure)
  5. Activate the new firmware (reboot, swap partitions, set boot flag — your choice)

Minimal example:

void fotaStartInstalltation(const void *xpFotaMetadata,
size_t xFotaMetadataLen,
const TTargetComponent *xpComponents)
{
if (!xpFotaMetadata || !xFotaMetadataLen || !xpComponents)
return;
// Copy the download URL
char url[512] = {0};
memcpy(url, xpComponents->componentUrl, xpComponents->componentUrlLen);
// >>> YOUR PLATFORM CODE HERE <<<
// 1. Download firmware from `url` over HTTPS
// 2. Write to your flash/update area
// 3. Validate the image
// 4. Call fotaUpdateComponent() — see Section 3.3
// 5. Activate new firmware (reboot, swap, etc.)
}

3.3 <tt>fotaUpdateComponent()</tt> — You Call This

After your download + flash write finishes, you must call this to tell KTA the result.

TKFotaStatus fotaUpdateComponent(const uint8_t *componentName,
size_t componentNameLen,
const uint8_t *componentVersion,
size_t componentVersionLen,
TFotaState state);
// Success — pass the NEW version:
fotaUpdateComponent(name, nameLen, newVersion, newVersionLen, E_K_FOTA_SUCCESS);
// Failure — pass the OLD version:
fotaUpdateComponent(name, nameLen, oldVersion, oldVersionLen, E_K_FOTA_ERROR);

4. FOTA Storage (Persistence)

KTA needs to save campaign state so it can resume after a reset. You provide two functions that read/write raw bytes by ID to whatever NVM your MCU has.

Edit: SOURCE/salapi/k_sal_fotastorage.c

bool salFotaStorageWrite(uint32_t xStorageDataId, const uint8_t *xpData, size_t xDataLen);
bool salFotaStorageRead(uint32_t xStorageDataId, uint8_t *xpData, size_t *xpDataLen);

Storage IDs KTA uses:

ID Value Max Size Purpose
FOTA_STORAGE_NAME_ID 0x2001 36 bytes Campaign name
FOTA_STORAGE_METADATA_ID 0x2003 68 bytes Campaign metadata
FOTA_STORAGE_COMMIT_ID 0x2005 12 bytes Commit flag (written last for power safety)
FOTA_STORAGE_COMPONENT_ID(i) 0x2010 + i ~558 bytes Component record (name, version, URL, state)

Rules:

  1. Each ID maps to a separate NVM region — regions must not overlap.
  2. Read must return exactly what was last written for the same ID.
  3. Data must survive reset.
  4. If nothing was written for an ID, return false.
  5. You do not interpret the data — just store and retrieve raw bytes.

If your MCU has no NVM (or for quick evaluation), use this stub:

bool salFotaStorageWrite(uint32_t id, const uint8_t *data, size_t len)
{
return true; // KTA keeps state in RAM
}
bool salFotaStorageRead(uint32_t id, uint8_t *data, size_t *len)
{
return false; // No saved state — KTA starts fresh
}

With the stub, FOTA cannot resume after power loss — the download must complete in one session.

5. KTA Agent Integration

Your application must call the KTA agent periodically so it can communicate with keySTREAM and receive FOTA campaigns.

// 1. After network is ready — initialize once:
ktaKeyStreamInit();
// 2. In your main loop — call every ~5 seconds:
TKktaKeyStreamStatus status;
ktaKeyStreamFieldMgmt(true, &status);
// 3. If KTA says to re-init:
if (status == E_K_KTA_KS_STATUS_REFURBISH)
ktaKeyStreamInit();

When a campaign arrives, KTA automatically:

  1. Calls your fotaPlatformGetComponents() to check current version
  2. Saves state via your salFotaStorageWrite()
  3. Calls your fotaStartInstalltation() with the download URL

6. Bootloader

You must provide your own bootloader. Its job:

  1. Verify the firmware image (hash, signature, CRC — your choice)
  2. Boot if valid — jump to application
  3. Roll back if invalid — boot previous firmware or enter recovery

How you implement this depends entirely on your MCU architecture. The SG41 reference uses ECC608 for ECDSA signature verification and dual-bank swap — study firmware/sg41_boot/src/app.c for one approach.

7. Integration Checklist

  • [ ] **fotaPlatformGetComponents()** — returns your firmware name + version
  • [ ] **fotaStartInstalltation()** — downloads from URL, writes to flash, calls fotaUpdateComponent()
  • [ ] **salFotaStorageWrite/Read()** — stores/loads FOTA state in your NVM (or use stub)
  • [ ] KTA agent loop — ktaKeyStreamInit() + periodic ktaKeyStreamFieldMgmt()
  • [ ] Bootloader — verifies new image, jumps or rolls back
  • [ ] Config — set KEYSTREAM_DEVICE_PUBLIC_PROFILE_UID to match your ECC608 fleet profile
  • [ ] Test — create a campaign in TPDS, verify download → reboot → new version reported

8. Troubleshooting

Symptom Fix
KTA reports wrong version Check fotaPlatformGetComponents() — is it reading the real running version?
Download never starts Check fotaStartInstalltation() — are you triggering your download logic?
Download fails Check your TLS/network stack and CA certificate
Reboot shows old version Check that you activate the new firmware after fotaUpdateComponent()
FOTA state lost after reset Check salFotaStorageWrite() — is it actually writing to NVM?
Campaign not received Is ktaKeyStreamFieldMgmt() being called every ~5 seconds?
